## 1. Executive Summary

`zupfnoter-ts` is a well-structured TypeScript monorepo that rewrites the Zupfnoter
harp-notation generator. The core ABC → Song → Sheet → SVG transformation pipeline is
largely implemented and **151 of 161 tests pass** (10 sheet-comparison tests are
intentionally skipped because legacy fixtures have not yet been populated).

The architecture is clean, the type system is strict, and pure-function style is
consistently applied. Several stubs and logic gaps left from early phase work should be
addressed before Phase 5 (Vue frontend) begins.

## 3. Strengths

### Architecture

- Clear monorepo separation: `@zupfnoter/types` (no logic), `@zupfnoter/core` (no DOM),
  `SvgEngine` (string output only). Each package is independently buildable and testable.
- `Confstack` is correctly isolated from domain knowledge. The Zupfnoter-specific stack
  assembly lives in `buildConfstack.ts` (a separate adapter), keeping `Confstack.ts` generic.
- `AbcModel` is an internal module — only `AbcParser.ts` imports it. The public API of
  `@zupfnoter/core` never leaks abc2svg internals.
- Circular references (`prevPlayable` / `nextPlayable`) are explicitly documented as
  non-serializable and are excluded from JSON output via `songToFixture()`.

### Code quality

- `strict: true` + `noUncheckedIndexedAccess: true` in `tsconfig.base.json` — the strongest
  TypeScript safety settings available.
- All source files carry JSDoc headers referencing the legacy Ruby equivalent
  (`harpnotes.rb`, `confstack.rb`, etc.), making the port traceable line-by-line.
- Immutable discriminant fields (`readonly type: 'Note'`) on every union member enable
  exhaustive type narrowing.
- Pure functions throughout: `pitchToX`, `beatToY`, `durationToKey`, all BeatPacker methods.
  No hidden global state.

### Testing

- All five BeatPacker pack methods have dedicated tests.
- `semanticMatch.ts` applies toleranced floating-point comparison for positions (±0.1 mm)
  and sizes (±0.05 mm) — appropriate for a graphics-layout pipeline.
- `dump_ts_output.spec.ts` writes JSON fixtures as a dev-helper: run once, inspect, commit.
- Snapshot tests (`HarpnotesLayout.spec.ts`) catch regressions across the full
  ABC → Song → Sheet path.

## 4. Issues

### 4.1 Type-check failure – `semanticMatch.ts` (8 errors)

**Severity: blocker** — `pnpm type-check` exits with status 2, which would fail CI.

```
semanticMatch.ts(239,18): error TS18048: 'ac' is possibly 'undefined'.
semanticMatch.ts(239,29): error TS18048: 'fc' is possibly 'undefined'.
... (8 errors total, lines 239–253)
```

Under `noUncheckedIndexedAccess`, array element accesses (`ac.size[0]`, `fc.from[0]`,
etc.) can be `undefined`. The comparisons at lines 239–253 need null-guards such as:

```ts
// before
!nearlyEqual(ac.size[0], fc.size[0], SIZE_TOLERANCE)

// after
const acSize = ac.size
const fcSize = fc.size
if (fcSize !== undefined) {
  if (acSize === undefined) { ... }
  else if (!nearlyEqual(acSize[0] ?? 0, fcSize[0] ?? 0, SIZE_TOLERANCE) || ...) { ... }
}
```

### 4.2 `_charposToLineCol` – permanent stub

**Severity: medium** — affects every entity in every `Song`.

```ts
// Phase 2: return placeholder — full implementation needs the ABC source text
return [1, 1]
```

Every `startPos` and `endPos` on every entity is `[1, 1]`. This makes bidirectional
editor ↔ SVG selection (Phase 5) impossible until resolved.

**Fix:** Pass the ABC source string from `AbcParser` into `AbcToSong.transform()`, then
implement a character-offset → `[line, col]` lookup (pre-build a cumulative newline
position array on parse).

---

### 4.3 `_parseDecorations` – permanent stub

**Severity: low** — decorations never appear in the Song.

```ts
// Phase 2: return empty array
return []
```

The `decoration.abc` fixture test passes vacuously. abc2svg stores decorations in
`sym.extra` entries with known type IDs. The legacy Ruby port checks for specific
`extra.type` values corresponding to fermata, emphasis, etc.

---

### 4.4 `_parseTuplet` – `tupletEnd` always `false`

**Severity: medium** — tuplet brackets are never rendered.

```ts
return { tuplet: tplet.p ?? 3, tupletStart: true, tupletEnd: false }
```

`HarpnotesLayout._layoutVoiceTuplets()` draws the bracket only when `p.tupletEnd` is
`true`. Since it is hardcoded `false`, no tuplet brackets appear in the SVG output.

**Fix:** After `tupletStart` is detected, count `p` notes (the tuplet group size from
`tplet.p`) and set `tupletEnd: true` on the last note of the group. State tracking in
`VoiceState` would be the cleanest approach (similar to how ties are tracked).

---

### 4.5 Volta variant logic incomplete

**Severity: medium** — variant colouring is wrong for first endings.

```ts
// Volta bracket start
if (sym.rbstart === 2) {
  state.variantNo = Math.min(state.variantNo + 1, 2) as 0 | 1 | 2
}
```

Only `rbstart === 2` (second ending) updates `variantNo`. The expected behaviour:

| Event | Action |
|-------|--------|
| `rbstart === 1` | `variantNo = 1` |
| `rbstart === 2` | `variantNo = 2` |
| After volta bracket closes (`rbstop`) | `variantNo = 0` |

The current code leaves first-ending notes coloured as variant 0 (default), and never
resets the variant after the bracket closes.

---

### 4.6 Chord symbol detection – fragile heuristic

**Severity: low** — edge-case misclassification.

```ts
// Chord symbol: starts with uppercase letter (heuristic)
if (/^[A-G]/.test(text)) { /* chord */ } else { /* annotation */ }
```

Any annotation whose text starts with A–G (e.g. `"Good luck"`, `"D.C. al Fine"`) would
be misclassified as a chord symbol. The legacy system uses `AbcExtra.type` (an integer
type ID) to distinguish chord symbols from free annotations. That field is already typed
on `AbcExtra` in `AbcModel.ts`.

---

### 4.7 Sheet legacy comparison tests are all skipped

**Severity: medium** — no regression validation for Layout → Sheet against legacy output.

```
↓ src/testing/__tests__/sheet/legacy_comparison.spec.ts (10 tests | 10 skipped)
```

`fixtures/sheet/*.json` files exist but contain empty `children: []` placeholders. The
legacy Ruby export (`tools/legacy-song-to-fixture.mjs`) has not been run against a Ruby
Zupfnoter instance to populate them. Until done, layout regressions are undetected.

---

### 4.8 `packages/core` has no `lint` script

**Severity: low** — oxlint is silently skipped on core source files.

`pnpm -r run lint` delegates to each package's `lint` script. `packages/core/package.json`
has no `lint` script, so the most logic-heavy package is never linted.

**Fix:** Add to `packages/core/package.json`:

```json
"lint": "oxlint ."
```

---

### 4.9 `BeatCompressionMap` uses object keys (string coercion)

**Severity: low** — potential silent precision loss.

```ts
export type BeatCompressionMap = Record<number, number>
```

JavaScript object keys are always coerced to strings. When `beat` is a non-integer (rare
but possible with fractional abc2svg time values), lookups like `beatMap[beat]` silently
match or miss based on string representation. Using `Map<number, number>` would be
type-safe and unambiguous.
